# Security Audit — Taruu

_Static (SAST) audit, 36 parallel agents. 30 raw → 22 confirmed, 4 needs-context, 4 false-positive. Each finding adversarially verified. No live target — pre-deploy code audit. 2026-06-16._

## Confirmed (severity-ranked)

### 1. [HIGH] Treasury deposit RPC has no idempotency guard — double-credit of municipality balance on multi-event Paddle delivery
- **File:** `supabase/migrations/20250116000001_treasury_and_issue_coins.sql:257-293; apps/web/src/app/api/payments/webhook/route.ts:105-144; apps/web/src/lib/supabase/db.ts:1221-1244`
- **Issue:** record_treasury_deposit() unconditionally does `balance_ils = balance_ils + p_amount_ils` and INSERTs a new treasury_transactions row. There is NO unique constraint on treasury_transactions.payment_id (schema lines 64-85 only have non-unique indexes), so the same payment can credit the treasury ledger more than once. The webhook's only protection against re-running fulfilment is (a) the webhook_events.event_id UNIQUE gate and (b) the in-route payment.status === 'completed' check at route line 105. The event_id gate does NOT help here: parseWebhookEvent (services/payments/paddle.ts:377-381) maps BOTH `transaction.completed` AND `transaction.paid` to the same internal `payment.succeeded` type, and Paddle emits these as two DISTINCT events with two DISTINCT event_ids for one transaction. Both pass the event_id dedup. The only remaining guard is the payment.status check, but updatePaymentStatus (db.ts:535-555) is a plain `update().eq('id')` with NO status precondition — so it is a non-atomic check-then-set. Two near-simultaneous deliveries both read status='pending', both proceed past line 105, both call recordTreasuryDeposit → balance_ils credited twice; the same window also double-mints SYNC tokens (route 157-167, no on-chain idempotency) and double-creates the entitlement (route 147-154). recordUserVote is the only step protected (user_votes UNIQUE(user_id,vote_id), db.ts/schema:240), so the vote tally survives, but money/token state does not.
- **Exploit:** Attacker is not even required — normal Paddle behaviour triggers it: a single successful checkout fires transaction.completed and transaction.paid. If they land concurrently (Paddle delivers webhooks in parallel and retries on any non-2xx), both pass the event_id gate (different ids) and the non-atomic status check, crediting the per-vote/municipality treasury ledger twice and minting double SYNC tokens for one ₪3/₪50 payment. Since the treasury ledger is what gets batch-seeded into the Bags.fm bag at resolution, inflated balances translate into real SOL/liquidity being over-allocated. An attacker who can replay a captured (still signature-valid, <5min old) delivery can amplify this on demand.
- **Fix:** Make fulfilment idempotent at the data layer, not via a TOCTOU status read: (1) add `ALTER TABLE treasury_transactions ADD CONSTRAINT uq_treasury_deposit_payment UNIQUE (payment_id, type)` and have record_treasury_deposit do `INSERT ... ON CONFLICT (payment_id, type) DO NOTHING` and only add to balance_ils when a row was actually inserted (check ROW_COUNT). (2) Make the pending→completed transition atomic and the gate for all downstream work: change updatePaymentStatus to `update({status:'completed'}).eq('id',id).eq('status','pending').select().maybeSingle()` and only run treasury/mint/entitlement when a row was returned (mirrors the correct markMerchOrderPaid pattern at db.ts:2002-2023). (3) Optionally collapse transaction.paid handling so only one Paddle event type triggers fulfilment.

### 2. [HIGH] GPS coordinates accepted verbatim and never verified at vote time — trivial location spoofing
- **File:** `apps/web/src/app/api/votes/[id]/participate/route.ts:165`
- **Issue:** participate accepts gpsCoordinates from the request body and only base64-encodes {lat,lng,timestamp} into a 'locationHash' for the blockchain record. It never calls verifyLocationInMunicipality / verifyCheckIn against the vote's municipality. The dedicated /api/votes/[id]/verify-location endpoint exists but is advisory only — its result is not required before participate. There is no device attestation, no signed-location channel, and timestamp is attacker-supplied.
- **Exploit:** Attacker POSTs participate with gpsCoordinates:{latitude:32.0853,longitude:34.7818,timestamp:Date.now()} for any municipality regardless of their real location. The vote is recorded as if cast inside Tel Aviv. A resident of one city (or another country) can vote in any municipality's polls.
- **Fix:** At participate time, re-run verifyCheckIn(latitude, longitude, accuracy, vote.municipality_id) server-side and reject if not inside bounds and accuracy unacceptable. Treat the client-supplied timestamp as untrusted (use server time / a short freshness window). Recognize that browser geolocation is inherently spoofable; for a financial+civic system consider stronger signals, but at minimum enforce the bounds check server-side instead of recording unverified coordinates.

### 3. [MEDIUM] Google OAuth callback exchanges any code with no server-side state/PKCE — login CSRF / code injection
- **File:** `apps/web/src/app/api/auth/callback/route.ts:26-49 (and services/auth/google.ts:42-58)`
- **Issue:** The primary Google sign-in callback POST /api/auth/callback reads only `code` from the body and immediately calls exchangeCodeForTokens(code,...) (lines 28-49). It never receives or validates the OAuth `state` parameter. The only CSRF protection (generateOAuthState/storeOAuthState/verifyOAuthState) lives entirely client-side and is backed by sessionStorage (google.ts:42-58) — verifyOAuthState reads `sessionStorage.getItem('oauth_state')` in the browser and returns false on the server, so it can never gate the server exchange. There is no PKCE (no code_challenge/code_verifier in buildGoogleAuthUrl, google.ts:70-78). Contrast with the social-connect callbacks, which DO verify a signed JWT state via verifyOAuthState() from lib/oauth-state.ts (e.g. social/callback/facebook/route.ts:58-83).
- **Exploit:** Classic OAuth login-CSRF: an attacker completes Google auth for their own account, captures the resulting `code`, and tricks a victim's browser into POSTing it to /api/auth/callback (e.g. an auto-submitting form / fetch on an attacker page; the endpoint takes JSON code with no anti-CSRF token and no state binding). The victim is silently logged into the ATTACKER's account; subsequent votes, payments, profile data and merch orders the victim creates accrue to the attacker-controlled account. Because there is no state, a stolen/leaked authorization code from any context can also be replayed into the exchange.
- **Fix:** Bind the OAuth flow to the initiator server-side: generate a signed state (reuse lib/oauth-state.createOAuthState) at sign-in initiation, store it in an HttpOnly cookie, include it in the Google auth URL, and in /api/auth/callback require + verify the returned `state` against the cookie (and verifyOAuthState signature) before exchanging the code. Strongly prefer adding PKCE (code_challenge S256 at init, code_verifier at exchange). Do not rely on the client-side sessionStorage check for any security guarantee.

### 4. [MEDIUM] SYNC token minting and entitlement creation are not idempotent — double-mint on webhook re-fire
- **File:** `apps/web/src/app/api/payments/webhook/route.ts:147-170`
- **Issue:** On payment.succeeded the webhook calls createEntitlement (147) and qubikService.mintTokens (159) with no dedup key tied to the payment. Like the treasury issue, the only thing preventing a second execution is the non-atomic payment.status check at line 105 + updatePaymentStatus (db.ts:535, no status precondition). Because transaction.completed and transaction.paid arrive as separate event_ids and updatePaymentStatus is a blind write, a concurrent/duplicate delivery mints tokens twice and writes a duplicate entitlement row (entitlements has no unique (payment_id) constraint enforcing one grant per payment).
- **Exploit:** Same trigger as the treasury finding: the natural two-event Paddle delivery, or a replayed still-valid signed delivery, races the status check and mints SYNC twice for one purchase. 1 ILS = 1 SYNC, so a creation payment (₪50) can yield 100 SYNC instead of 50. Tokens are a tradeable asset, so this is direct value inflation.
- **Fix:** Gate minting/entitlement on the atomic pending→completed transition (return value of a status-guarded update) so only the delivery that actually flips the row performs fulfilment. Add a UNIQUE(payment_id) on entitlements for vote/create grants, and pass a deterministic idempotency key (e.g. payment.id) to qubikService.mintTokens so the mint layer can dedupe.

### 5. [MEDIUM] Green Invoice & Printful webhook shared secrets are placed in the URL query string (?token=<secret>)
- **File:** `apps/web/src/app/api/merch/checkout/route.ts:160-167, apps/web/src/app/api/merch/webhook/route.ts:21,42, apps/web/src/app/api/merch/fulfillment-webhook/route.ts:5-7,35-37`
- **Issue:** The merch checkout registers the notify URL with the full webhook secret embedded as a query parameter: `${origin}/api/merch/webhook?token=${encodeURIComponent(webhookSecret)}` (checkout/route.ts:162). Both webhook handlers then read the secret from `new URL(request.url).searchParams.get('token')`. Secrets in URLs are a classic leakage vector: query strings land in web-server / proxy / CDN access logs, browser history, Referer headers, and third-party (Green Invoice / Printful) dashboards and audit logs that show the configured notify URL verbatim. Cloudflare Workers observability is enabled (wrangler.jsonc `observability.enabled: true`), so the request URL — including ?token= — is captured in Workers logs. Unlike the cron routes (which correctly use an Authorization: Bearer header, resolve-votes/route.ts:23) and Paddle (Paddle-Signature header), these two webhooks lean on a URL token as the primary auth path. The constant-time compare itself is implemented correctly (length guard before timingSafeEqual), so the weakness is purely the transport/storage of the secret, not the comparison.
- **Exploit:** Anyone with read access to Workers logs, the upstream CDN/proxy logs, or the Green Invoice/Printful webhook configuration screen can read GREENINVOICE_WEBHOOK_SECRET / PRINTFUL_WEBHOOK_SECRET in cleartext. With that token an attacker POSTs a forged `{ custom: <orderId> }` body to /api/merch/webhook?token=<leaked> and flips an arbitrary pending order to 'paid' (markMerchOrderPaid) and triggers Printful fulfilment — obtaining physical merch without paying. The fulfillment-webhook token similarly lets an attacker mark orders shipped with bogus tracking.
- **Fix:** Stop putting the secret in the URL. Prefer a signed-body HMAC like the Paddle path (sign the raw body, send hex in a header, verify with createHmac + length-guarded timingSafeEqual). If Green Invoice/Printful only support a static shared secret, send it exclusively via the header the code already accepts (x-greeninvoice-token / x-printful-token) and remove the ?token= from the registered notify URL. Also scrub query strings from Workers logs.

### 6. [MEDIUM] Treasury transactions endpoint returns every user's transactions (incl. their user_id) to any authenticated user
- **File:** `apps/web/src/app/api/treasury/[municipality]/transactions/route.ts:66-86`
- **Issue:** The route doc-comment claims 'users can only see transactions they initiated, or all if admin.' In reality it calls getTreasuryTransactions(treasury.id, {limit, offset, type}) with NO user filter (lib/supabase/db.ts:1175-1190) and runs under the service-role client, which bypasses the RLS policy 'Users can see their own treasury transactions' (USING user_id = auth.uid()). It then maps and returns userId for every transaction (line 78). Any logged-in user can therefore enumerate the full deposit/allocation/withdrawal ledger for a municipality, including which user_id made each ₪ deposit and the exact amount_ils — a privacy leak that links real users to their civic spending. The RLS policy that was supposed to prevent this is silently bypassed because the service role is used.
- **Exploit:** Log in as any user, GET /api/treasury/tel-aviv/transactions?limit=100&offset=0 repeatedly. Receive a paginated dump of every deposit with user_id + amount, deanonymizing who funded which vote and how much. The 'admin only' gate the comment promises does not exist.
- **Fix:** Either (a) scope the query to the caller: pass session.userId and add .eq('user_id', userId) in getTreasuryTransactions for non-admin callers, or (b) strip user_id from the response and only expose aggregate/anonymized fields publicly. Do not rely on the RLS policy here since the service-role client bypasses it.

### 7. [LOW] Bags.fm swap executes a fully client-supplied quote with no server-side re-derivation
- **File:** `apps/web/src/app/api/bags/swap/route.ts:35-71; apps/web/src/services/bags/index.ts:215-241`
- **Issue:** POST /api/bags/swap takes `quote` straight from the request body and forwards it verbatim to bagsService.executeSwap → Bags.fm /trade/swap. The route only checks that quote.inputAmount/outputAmount/fee are present (lines 47-52); it never re-fetches the quote server-side, never validates that the quote was one this server issued, and never binds it to the prior /api/bags/quote response. inputAmount, outputAmount and fee are entirely attacker-controlled.
- **Exploit:** A signed-in user posts a hand-crafted quote object (e.g. understated `fee`, or output/input ratios that were never quoted) to /api/bags/swap. Whether this drains value depends on how Bags.fm validates the quote server-side: if Bags re-validates against live pool state the blast radius is limited to the caller's own wallet; if it trusts the quote fields (e.g. for fee accounting on platform-fee-share positions), a tampered fee can shortchange the platform/municipality fee split (createDefaultFeeShareConfig routes 80% to the municipality treasury). At minimum it removes the server's ability to enforce slippage/fee policy.
- **Fix:** Never trust a client quote for execution. Either (a) have /api/bags/quote persist the issued quote (server-side, short TTL, bound to the session) and have /swap accept only a quote id it looks up and re-validates, or (b) have /swap re-call bagsService.getQuote with the same input/output mints+amount and execute the freshly fetched quote, rejecting if the client's numbers deviate beyond tolerance.

### 8. [LOW] Cron secret compared with non-constant-time string equality
- **File:** `apps/web/src/app/api/cron/resolve-votes/route.ts:35, apps/web/src/app/api/cron/verification-notifications/route.ts:38, apps/web/src/app/api/cron/mint-nfts/route.ts:25`
- **Issue:** All three cron routes authenticate with `if (authHeader !== \`Bearer ${CRON_SECRET}\`)` — a plain JS `!==` string comparison, which short-circuits on the first differing byte and is not timing-safe. By contrast the Paddle and merch webhook secrets are compared with crypto.timingSafeEqual. The cron secret guards privileged, state-mutating endpoints (vote resolution, NFT minting, push notifications).
- **Exploit:** In principle an attacker could mount a remote timing side-channel to recover CRON_SECRET byte-by-byte, then call the cron endpoints to force-resolve votes / trigger minting. In practice this is low severity: network jitter over the public internet makes byte-level timing recovery against a short-circuit string compare extremely difficult, and the secret is high-entropy. Still inconsistent with the codebase's own standard.
- **Fix:** Compare with a constant-time check: decode both sides to Buffers, length-guard, then crypto.timingSafeEqual — exactly as merch/webhook/route.ts already does. Extract a shared `timingSafeStringEqual` helper and use it in all three cron routes.

### 9. [LOW] Bags webhook signature verifier will throw on length mismatch (latent), and compares raw bytes vs hex
- **File:** `apps/web/src/services/bags/index.ts:286-306`
- **Issue:** verifyWebhookSignature does `crypto.timingSafeEqual(Buffer.from(signature), Buffer.from(expectedSignature))` with no length guard. Node's timingSafeEqual throws a RangeError when the two buffers differ in length, so any attacker-supplied signature whose length != 64 (the hex digest length) raises an uncaught exception instead of returning false. It also compares the caller's raw signature string bytes against a hex-encoded expected digest, so a correctly-formed HMAC supplied as raw bytes would never match. This function is exported from bagsService but is currently NOT wired to any route (apps/web/src/app/api/bags/ contains only quote, swap, trending — no webhook), so it is dead/latent today; hence low/info rather than high.
- **Exploit:** Not exploitable today because no route invokes it. If a bags webhook route is later added using this verifier as-is, a forged request would either crash the handler (DoS / 500) or — depending on how the caller handles the throw — could be mishandled. The hex-vs-raw mismatch would also break legitimate verification.
- **Fix:** Mirror the correct Paddle implementation: `const a = Buffer.from(sig, 'hex'); const b = Buffer.from(expected, 'hex'); if (a.length !== b.length) return false; return timingSafeEqual(a, b);` Add the length guard and decode both sides from the same encoding before wiring any webhook route to it.

### 10. [LOW] HTML injection in refund-request support email (user reason interpolated raw)
- **File:** `apps/web/src/services/email/index.ts:186,196 (source: apps/web/src/app/api/payments/refund/route.ts:37,60,92)`
- **Issue:** sendRefundRequestNotification builds the support email HTML by interpolating the user-supplied refund `reason` (and `userEmail`/`userId`) directly into a template literal with zero HTML escaping: line 186 `...<td>${k}</td><td>${v}</td>...` where v includes `params.reason`, then injected into the body at line 196 `<table>${rows}</table>`. The refund route (refund/route.ts:37) only does `.trim()` and a length<=500 check — no sanitization — before passing it to the email (line 92). There is no escapeHtml/DOMPurify helper anywhere in the codebase (grep returned nothing).
- **Exploit:** Any authenticated user submits POST /api/payments/refund with reason containing HTML, e.g. `<img src=x onerror=...>` or `<a href="https://evil.tld">click</a>` or a hidden <style>/<div> that spoofs the email body. The email is sent to support@taruu.co.il with replyTo set to the attacker's address — a classic phishing/content-spoofing primitive against internal support staff (and stored-HTML rendering in whatever mail client/helpdesk ingests it). 500 chars is ample for a working payload.
- **Fix:** Add a small escapeHtml() utility (replace & < > " ') and apply it to every interpolated user value (reason, userEmail, userId, paymentId, providerId) in the HTML branch. Keep the plain-text branch as-is. Prefer rendering structured values rather than concatenating into raw HTML.

### 11. [LOW] Stored XSS via vote title/description and user firstName/lastName in transactional emails
- **File:** `apps/web/src/services/email/index.ts:227,292,299,352,359,418,425,433,487 (sinks); apps/web/src/app/api/votes/route.ts:159,202; apps/web/src/app/api/user/profile/route.ts:139-145 (sources)`
- **Issue:** getWelcomeTemplate, getVoteNotificationTemplate, getVoteCreatedTemplate and getVoteResultsTemplate interpolate `firstName`, `voteTitle`, `municipality`, `winningOption`, `userVotedFor` directly into HTML email bodies with no escaping (e.g. line 227 `שלום ${firstName}`, line 299 `<h3>...${params.voteTitle}</h3>`). The sources are user-controlled and stored unsanitized: profile firstName/lastName are read straight from the request body (profile/route.ts:139, only presence-checked) and vote title/description from votes POST body (votes/route.ts:111 only checks presence). The stored values are later blasted to other residents (vote notification broadcast, votes/route.ts:199-208) and to the creator.
- **Exploit:** An attacker sets their profile firstName to `<img src=x onerror=...>` or creates a vote whose title contains HTML. When the welcome/vote-created/vote-notification email renders, the markup executes/renders in recipients' mail clients. Because vote-notification fan-outs to every other resident of the municipality (sendInBatches over getUsersByMunicipality), one malicious vote title becomes stored content-injection delivered to many third-party inboxes — beyond simple spoofing this enables phishing links rendered as if from Taruu.
- **Fix:** HTML-escape all user-derived interpolations in every template (firstName, lastName, voteTitle, description, municipality, winningOption, userVotedFor). Centralize via an escapeHtml helper applied at the template layer so new templates inherit it.

### 12. [LOW] OTP attempt lockout is bypassable by re-sending a code (counter reset on every send)
- **File:** `apps/web/src/app/api/user/phone/send-code/route.ts:188`
- **Issue:** The verify route's brute-force gate reads `attempts` from phone_verifications (DB, keyed by user_id) and blocks at >=5 in 10 min. But send-code resets `attempts: 0` every time a code is issued (insert path line 188-196 and update path line 202-210), and the in-app OTP store (services/sms/otp.ts MAX_ATTEMPTS=5, keyed by phone in KV) is also re-created fresh on each send. So the 5-attempt cap is per-code, not per-phone-lifetime. An attacker can interleave send→5 guesses→send→5 guesses indefinitely. With a 6-digit code (1e6 space) and a 600s TTL, repeated re-sends turn a bounded guess into an open-ended online brute force.
- **Exploit:** For a target phone the attacker controls a session for: call send-code, submit 5 wrong codes, call send-code again (resets attempts to 0 and issues a new code/hash), repeat. The only ceiling is the send-rate-limit (3/hr, 5/day) — but see the separate finding that this limit fails open and resets, and note each send yields a fresh code so even within limits the per-code 5-guess cap never accumulates into a real lockout.
- **Fix:** Track verification attempts cumulatively across re-sends within a verification session (do not zero `attempts` on send; only reset after a successful verify or a long cooldown). Enforce a hard per-phone ceiling (e.g. max 10 total guesses and max N sends per rolling window) that survives re-sends, and lock the phone for a cooldown when exceeded. Keep the attempt counter and the OTP record on the same key (see key-mismatch finding).

### 13. [LOW] send-code rate-limit reads a single DB row and continues on any DB error (fail-open)
- **File:** `apps/web/src/app/api/user/phone/send-code/route.ts:214`
- **Issue:** The per-user send limits (3/hr, 5/day) are derived from a single phone_verifications row read; if the read errors or the row is absent the limit checks are skipped, and the subsequent attempt-recording is wrapped in try/catch that explicitly logs and 'Continue anyway - SMS was already sent' (line 216). There is no IP-based limit and no Upstash limiter on this endpoint (unlike check-in/participate which use the Redis limiter), so the only throttle is this best-effort DB counter that fails open.
- **Exploit:** If the phone_verifications write/read path errors (DB hiccup, type-assertion edge, missing row), send-attempt accounting is skipped and the caller can request many SMS messages — an SMS-pumping / toll-fraud and cost-amplification vector against the configured gateway, and it also feeds the OTP brute-force reset issue by allowing rapid fresh-code issuance.
- **Fix:** Move send-code throttling to the same Upstash sliding-window limiter used elsewhere (lib/rate-limit.ts), keyed by both user_id and phone (and an IP-based limiter), so it is independent of the DB row and fails closed. Do not proceed to send when the throttle store is unavailable; treat throttle-store failure as a soft deny rather than fail-open.

### 14. [LOW] merch_orders table has no RLS enabled or policies (defense-in-depth gap on PII)
- **File:** `supabase/migrations/20260615000003_merch_orders.sql:6-24 (and 20260616000001_merch_tracking.sql)`
- **Issue:** Every other user-data table runs ALTER TABLE ... ENABLE ROW LEVEL SECURITY (users, payments, push_tokens, phone_verifications, treasury_transactions, issue_coin_holdings, vote_nfts). merch_orders is the sole exception: the create migration and the tracking migration never enable RLS and define no policies. The table holds full shipping PII (fullName, email, phone, street, city, zip in the `shipping` JSONB) plus order items and amounts. Today all server access goes through the service-role client (lib/supabase/server.ts), so this is defense-in-depth rather than an active leak — the API route GET /api/merch/orders/[id] does enforce ownership (row.user_id !== session.userId -> 404). But with RLS off, the moment any query is run with the anon key (NEXT_PUBLIC_SUPABASE_ANON_KEY, shipped to the browser) the entire orders table — every buyer's name, address, phone, email — is world-readable, because Postgres defaults to allow when RLS is disabled.
- **Exploit:** If a future client-side feature (or a leaked anon key, which is by design public) ever calls supabase.from('merch_orders').select('*') with the anon key, an attacker dumps every customer's shipping address, phone and email. Unlike the other tables, there is no policy to stop it.
- **Fix:** Add to a migration: ALTER TABLE merch_orders ENABLE ROW LEVEL SECURITY; then a service-role full-access policy and (if any client read is ever needed) a SELECT policy USING (user_id = auth.user_id()). Match the pattern used by push_tokens / phone_verifications.

### 15. [LOW] Public votes listing runs unbounded queries (no LIMIT) — memory/DoS amplification
- **File:** `apps/web/src/lib/supabase/db.ts:710-752`
- **Issue:** getActiveVotes() and getVotesByMunicipality() build a select('*') ordered by created_at with NO .limit() / .range(). The public, unauthenticated GET /api/votes (apps/web/src/app/api/votes/route.ts:21-58) calls these directly and serializes the entire result set. There is no pagination parameter accepted at all. As the votes table grows (or is seeded with many ended/active rows per municipality), every hit to the homepage/listing pulls and serializes the whole table slice into the Worker's memory.
- **Exploit:** An unauthenticated attacker hammers GET /api/votes (and /api/votes?municipality=X) concurrently. Because each request loads the full unbounded result set into the Cloudflare Worker and there is no rate limit on this route, response size and per-request memory scale with table size, enabling cheap memory-pressure / cost-amplification DoS. Contrast with treasury transactions and holders endpoints which correctly cap at 100/500.
- **Fix:** Add a server-enforced cap to getActiveVotes/getVotesByMunicipality (e.g. .range(offset, offset+limit-1) with limit defaulting to 50, max 100) and accept limit/offset query params in /api/votes, mirroring the treasury/holders routes.

### 16. [LOW] Issue-coin holders endpoint exposes full wallet addresses and exact ILS invested despite 'anonymized' claim
- **File:** `apps/web/src/app/api/votes/[id]/issue-coin/holders/route.ts:66-82`
- **Issue:** The handler comment says 'Personal information is anonymized for non-authenticated requests.' It computes a truncated displayName, but then also returns the FULL untruncated walletAddress (line 75) and the exact investedILS (line 77) for every holder, to anyone, unauthenticated, up to 500 per page with working offset pagination (hasMore). For internal users it returns initials, but each external supporter's complete Solana wallet plus exact ILS contribution is public and fully enumerable.
- **Exploit:** Unauthenticated: GET /api/votes/{id}/issue-coin/holders?limit=500&offset=0... to scrape the complete cap table of every civic issue coin — full wallet addresses tied to exact ILS amounts invested in specific (often politically sensitive) municipal votes. On-chain analysis can then deanonymize backers.
- **Fix:** Drop walletAddress to the same truncated form already used for displayName, and bucket/round investedILS (or omit it) for the public response. Only return full wallet + exact amount to the authenticated owner of that holding.

### 17. [LOW] RLS policies on newer tables use auth.uid() but the app sets app.current_user_id — every per-user policy silently never matches
- **File:** `supabase/migrations/20250116000001_treasury_and_issue_coins.sql:191,203; 20250118000001_vote_nfts.sql; 20250119000001_phone_verifications.sql:103`
- **Issue:** The custom JWT auth helper auth.user_id() (20240101000001_rls_policies.sql:10-21) reads request.jwt.claims->>'sub' or app.current_user_id, and withUserContext() (lib/supabase/server.ts:35-37) sets app.current_user_id via set_claim. The original tables (users, payments, push_tokens) correctly use auth.user_id() in their policies. But the treasury, issue_coin_holdings, vote_nfts and phone_verifications migrations use Supabase's built-in auth.uid() instead. Since this project does not use Supabase Auth sessions (persistSession:false, custom jose JWTs), auth.uid() returns NULL under the anon role, so policies like 'Users can see their own treasury transactions' USING user_id = auth.uid() match NOTHING for a real user — and conversely, the 'publicly readable USING (true)' policies on treasury and issue_coins expose those tables to anon entirely. The inconsistency means the RLS layer does not behave as the comments claim; it only happens not to bite because all access is service-role today.
- **Exploit:** No direct exploit while only the service role is used, but it is a latent failure: any migration to client-side anon reads would (a) leak treasury and issue_coins to anon (USING true), and (b) break legitimate per-user reads (auth.uid() NULL) — the security model is not actually enforced where it's written.
- **Fix:** Standardize all per-user policies on auth.user_id() (the project's helper), not auth.uid(). Re-review the USING(true) SELECT policies on treasury/issue_coins/vote_nfts and decide intentionally what anon may read; do not leave them world-open by default.

### 18. [LOW] SMS send-code: no per-phone-number rate limit enforced in route + reset bug, enabling cross-number SMS cost abuse
- **File:** `apps/web/src/app/api/user/phone/send-code/route.ts:119-217`
- **Issue:** The route lets an authenticated user supply an arbitrary Israeli `phone` and triggers a real Twilio Verify SMS (a paid action). Rate limiting is a hand-rolled check on the user's phone_verifications row (send_attempts vs last_send_at), NOT the Upstash limiter used elsewhere, and it does NOT call the DB function can_send_phone_verification() that was written to enforce a per-phone-number hourly cap across users (20250119000001_phone_verifications.sql:147-151). So the only cap is per authenticated user (>=3/hr, >=5 lifetime since send_attempts is never reset by the time window). There is also a TOCTOU: the row is read, the SMS is sent, then the counter is written — concurrent requests can each pass the check before any increment. Combined with the lack of a per-phone limit, the same target number can be SMS-bombed by rotating accounts, and a single account can burst-send via the race.
- **Exploit:** Authenticated attacker scripts N concurrent POSTs to /api/user/phone/send-code with phone=<victim number>; the read-before-write race lets several SMS fire before send_attempts increments (Twilio cost + harassment of the victim's phone). Across multiple cheap Google-OAuth accounts the per-user cap is trivially multiplied against one target number, since the per-phone guard function is never invoked.
- **Fix:** Call can_send_phone_verification(user.id, phone) (the existing SECURITY function with the per-phone hourly check) instead of the ad-hoc JS check, and front the route with an Upstash limiter keyed on both userId and phone. Make the increment atomic (single UPSERT that returns the post-increment count and rejects if over limit) to close the TOCTOU.

### 19. [LOW] bags/swap forwards a client-supplied quote to the trading API using platform credentials with no server-side revalidation
- **File:** `apps/web/src/app/api/bags/swap/route.ts:35-71; apps/web/src/services/bags/index.ts:215-240`
- **Issue:** POST /api/bags/swap accepts a `quote` object straight from the request body, does only a shape check (inputAmount/outputAmount/fee present), and passes it verbatim to bagsService.executeSwap, which POSTs it to Bags.fm /trade/swap using the platform's API key and (if not supplied) the user's Qubik wallet. The server never re-fetches a fresh quote or verifies the supplied amounts/fee against the live market before executing. The comment even notes 'should only be called with a valid quote ... quotes expire' — but nothing enforces that the quote is genuine or unexpired server-side.
- **Exploit:** An authenticated user crafts a quote with a tiny inputAmount and inflated outputAmount (or zero fee) and submits it. Whether this succeeds depends on Bags.fm re-validating server-side; if Bags trusts the relayed quote (it was issued to this API key), the platform executes an off-market swap on its own funded path. At minimum it removes the platform's ability to enforce slippage/fee policy. Exploitability hinges on Bags' own validation, hence low/medium.
- **Fix:** Re-issue the quote server-side from the user's requested input token/amount immediately before executing (call /trade/quote with server-trusted params), ignore client-provided outputAmount/fee, and enforce a max-slippage bound. Treat the client quote as untrusted input.

### 21. [INFO] payments/create generates a time-based idempotency key, defeating idempotent retries
- **File:** `apps/web/src/app/api/payments/create/route.ts:77`
- **Issue:** When the client does not supply idempotencyKey, the server fabricates `${user.id}-${type}-${voteId||'create'}-${Date.now()}`. Because Date.now() is embedded, every retry of the same logical create request produces a different key, so getPaymentByIdempotencyKey (line 80) never matches and a fresh pending payment row + Paddle transaction is created each time. The idempotency mechanism is effectively a no-op unless the client remembers to send a stable key.
- **Exploit:** Not a direct money-drain (each payment still requires the user to complete a separate Paddle checkout), but a double-clicking or retrying user accumulates multiple pending payments/Paddle transactions for one intended vote, and the server-side idempotency guard the code advertises does not actually engage. It also enables a user to spin up unbounded pending payment rows.
- **Fix:** Derive the default key deterministically from the request semantics, not the clock — e.g. `${user.id}-${type}-${voteId||'create'}-${optionId||''}` (one in-flight create per user/vote/option), or require the client to pass an idempotencyKey and reject when absent for non-idempotent retries.

### 22. [INFO] Structured logger does not redact secrets/tokens/PII from context (defense-in-depth gap)
- **File:** `apps/web/src/lib/logger.ts:64-84`
- **Issue:** sanitizeContext() only normalizes Errors and verifies JSON-serializability; it performs NO key-based redaction. Any caller that passes an object containing an api key, token, signature, or PII field would log it verbatim (JSON in production, line 89). The current callers were audited and are clean — they log identifiers, statuses, counts, booleans (e.g. newsletter/route.ts:43 logs only `!!BEEHIIV_API_KEY`), and error objects, not raw secrets, and the bags/paddle error logs only include status + parsed error bodies. So this is not an active leak, but the logger provides no guardrail: one careless future `logger.error('x', { headers })` or `{ env: process.env }` would dump secrets into Workers logs.
- **Exploit:** No current exploit. Future-proofing: without a denylist, a single future logging mistake (logging request headers containing Authorization/cookies, or spreading process.env into context) would silently emit secrets into the production log aggregator, where anyone with log access could harvest them.
- **Fix:** Add a redaction pass in sanitizeContext that masks values whose keys match a denylist (authorization, cookie, token, secret, apikey, api_key, password, privatekey, private_key, jwt) and recursively walks nested objects, replacing matched values with '[REDACTED]'. This keeps the audited-clean call sites working while preventing accidental future leakage.
